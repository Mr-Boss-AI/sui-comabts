## 2. Two open polish items uncovered tonight

### A. "No character found" red flash on fresh zkLogin load before hydration

A user loading the app via a fresh zkLogin session sees a brief
red error toast ("No character found" / similar) before the
chain-check completes and the character renders. Root cause is
ordering: the `get_character` round-trip fires on auth_ok before
the `restore_character` flow has populated the server-side record.
The server's `handleGetCharacter` (`handler.ts:876-880`) sends
`sendError 'Character not found'` when `getCharacterByWallet`
returns null, the frontend dispatches `SET_ERROR` â†’ toast shows
for ~5s before the auto-fade kicks in.

**Fix shape (next session):** suppress the toast during the
`chain_check_pending` window. Either gate the SET_ERROR dispatch in
the error-handler on `state.authPhase !== "chain_check_pending"`,
or skip the early `get_character` call when `auth_ok` arrives with
`hasCharacter: false` (the rehydrate flow already sends
`restore_character` which returns the same payload as
`character_data` for our purposes â€” the get_character call is
redundant during a self-heal).

Not a SUI-loss vector; pure UX polish. Probably 30 min.

### B. Open Wagers list â€” can't inspect opponent's character/gear before Accept

The wager lobby renders `{creatorName} Lv.{level} {archetype} ({stats})`
but nothing else. The user has no way to see:
- Opponent's equipped loadout (weapon class, armour tier, accessories)
- Win/loss record + recent fights
- Whether their character is closer to mine in ELO than the
  matchmaking band suggests

**Fix shape (next session):** click on a row to open
`<OpponentInspector wallet={creatorWallet} />` modal that:
- Fetches opponent's on-chain Character NFT (we already have
  `fetchCharacterNFT` in `frontend/src/lib/sui-contracts.ts`).
- Reads equipped DOFs to render the loadout doll.
- Calls a new `GET /api/character/:walletAddress/wagers` for recent
  wager outcomes (or reuse `getFightHistory` data â€” server already
  has it).
- Surfaces a level-mismatch warning if `|me.level - opponent.level| > 2`.

Could reuse the `<PlayerProfileModal />` component pattern from
the tavern (`frontend/src/components/social/player-profile-modal.tsx`).

Not a SUI-loss vector; user-confidence UX. ~2-3 hours including
the inspector modal + server endpoint.

## 3. Deferred to mainnet prep â€” NOT a testnet bug

### zkLogin auto-signing without confirmation popups

Tonight's Phase A + Bug 5/6/8 commits deliberately left zkLogin
signing **silent** â€” every `signer.signAndExecuteTransaction` fires
the chain tx without an intermediate confirm modal. That's the
demo-headline smoothness ("one click, fight starts") and is the
right testnet UX.

**Before mainnet, this MUST change.** Logged in
[`MAINNET_PREP.md`](./MAINNET_PREP.md#ux-gates--zklogin-signing-must-require-explicit-confirmation)
under section D (Operational security) as a hard blocker. Summary:
- Every wallet-popup site needs an explicit `<ConfirmTxModal />`
  gate before the signer fires.
- Audit checklist of every current call site is in MAINNET_PREP.
- Shared modal shows: target function, amount + asset, recipient,
  gas estimate.
- A source-grep gauntlet should fail if a new `signAndExecuteTransaction`
  lands without a paired pre-popup gate.
- A "Trust this session for 30 minutes" opt-in toggle is fine; OFF
  by default.

Why this isn't a testnet bug: testnet SUI is free, the demo flow
is the headline, and an explicit popup before every action would
add 3-4 clicks per fight. On mainnet a walk-away or page-injection
attack could drain a wallet â€” confirms are the only defence depth.
