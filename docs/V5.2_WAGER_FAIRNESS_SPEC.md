## 1. Problem

`accept_wager` is currently first-come-first-served. Any wallet that
sees an open `WagerMatch` can call `accept_wager` with the matching
stake and the fight locks instantly:

```move
public fun accept_wager(wager, stake, registry, clock, ctx) { ... }
```

Two consequences:

1. **No level matching.** A Lv20 fully-geared wallet can accept a Lv4
   creator's 1 SUI wager and almost certainly win. The creator's
   stake becomes free SUI for whoever scripts the fastest accept loop.
2. **No creator say.** Even if the creator wanted to wait for a
   specific challenger (a friend, a known peer, someone they've
   scouted), they can't — anyone who clicks first wins the slot.

**Why a UI fix isn't enough.** The trustless guarantee of the wager
escrow is that the chain is the source of truth. If we gate accept
client-side only, a sophisticated attacker can craft a PTB and call
`arena::accept_wager` directly — bypassing the gate entirely. The
abuse path requires no special tooling: `sui client call` plus the
package ID is enough. **Both rules MUST be enforced in Move.**

## 2. Goals & non-goals

### Goals

- **G1.** A `request_accept_wager` call must abort unless the
  challenger's character level is within ±1 of the creator's level
  *at the moment the wager was created* (snapshot — see
  [§5 Struct changes](#5-struct-changes)).
- **G2.** Accepting no longer instantly locks the fight. A creator
  must explicitly approve the specific challenger before the wager
  moves to ACTIVE.
- **G3.** No funds may strand. Every escrow path must have a
  bounded-time exit; no transition leaves a player's stake parked
  with no way to recover it.
- **G4.** Existing trustless guarantees preserved: TREASURY can still
  admin-cancel and emergency-settle exactly as in v5.1.
- **G5.** **No referee-liveness gap.** TREASURY holds the AdminCap
  that decides winners — if the referee (server) is down,
  compromised, or simply stalls, an `ACTIVE` wager must not strand
  funds. Either participant has the on-chain right to reclaim the
  escrow after a bounded timeout, refund-only (no winner declared).
  Closes the "centralized referee = single point of failure" risk
  flagged by external review. See
  [§3.5 Referee-liveness escape hatch](#35-referee-liveness--the-escape-hatch).

### Non-goals

- Reputation, ELO bracketing, gear-score matching — out of scope for
  v5.2 fairness. Level ±1 is the minimum viable fairness gate. Wider
  matchmaking lives in a separate v5.3+ effort.
- Per-wallet block/allowlist — creator approval is per-challenge, not
  persistent. A creator who decided "I'll fight Alice" can change
  their mind for the next wager; no on-chain block list.
- Stake size brackets — already gated by the v5.1 server-side
  `MAX_WAGER_SUI_MIST` cap and `MIN_STAKE_SUI`. Out of scope here.

## 3. Design overview — the lock-on-request vs lock-on-approval trade-off

A natural intuition is **"lock the challenger's stake only when the
creator approves, so a slow-to-respond creator doesn't strand the
challenger's SUI."** Spelling out the constraint surfaces a problem:

> The challenger's coin can only be deposited by a transaction the
> challenger signs. The creator cannot author a transaction that
> consumes the challenger's `Coin<SUI>`. So a literal "lock on
> approve_challenger" would require a 3-transaction dance
> (request → approve → confirm-with-coin) where the challenger can
> back out at the confirm step. That is clunky UX and a worse race
> against attackers who can spam-confirm faster than the legitimate
> challenger.

The **recommended resolution** keeps the funds-locked-on-request
shape but gives the challenger a unilateral exit so they are never
actually stranded:

| Action | Who signs | Effect |
|---|---|---|
| `request_accept_wager(wager, stake, …)` | challenger | challenger's stake enters a separate `challenger_escrow` field; status → `PENDING_APPROVAL`; sets `pending_challenger` |
| `approve_challenger(wager, …)` | creator | `challenger_escrow` merges into main `escrow`; status → `ACTIVE`; fight starts |
| `decline_challenger(wager, …)` | creator | refunds `challenger_escrow` to challenger; clears pending fields; status → `WAITING` |
| `withdraw_challenge(wager, …)` | **challenger** | refunds `challenger_escrow` to challenger; clears pending fields; status → `WAITING`. Self-service exit — the challenger NEVER needs the creator to act in order to recover their stake. |
| `cancel_expired_challenge(wager, …)` | anyone | if `PENDING_APPROVAL` has been open for `CHALLENGE_TIMEOUT_MS`, refund + back to `WAITING`. Same anti-strand role as v5.1's `cancel_expired_wager`. |

This is "lock on request, but with unconditional unilateral
withdrawal" — strictly stronger than the user-recommended
"lock-on-approval" because the challenger keeps full control of their
funds *and* the creator never has to chase a confirmation.

**The escrow invariant** (proved by inspection of the transitions
table in [§7](#7-escrow-timing--failure-mode-table)): no transition
removes funds from either escrow except as a refund to the
original depositor or as a settlement payout per the existing v5.1
rules. The chain never burns SUI; the chain never sends SUI to a
third party.

## 3.5 Referee-liveness — the escape hatch

The wager system depends on a **centralized referee**: TREASURY
holds the AdminCap, runs the combat resolver, and signs `settle_wager`
/ `settle_tie` to release escrow to the winner. On the happy path
that's fine — fights resolve in seconds and the referee is just a
deterministic oracle for the combat output.

The unhappy path is **referee liveness failure**: server down,
treasury key rotation in flight, TREASURY-private-key compromised
and revoked, gRPC subscription stalled, or a chain congestion event
that delays settle txs past the SLA. In any of those, an `ACTIVE`
wager has both stakes locked in escrow with no on-chain path for
players to recover them. External review flagged this as the
single-point-of-failure risk the trustless escrow design is meant to
prevent.

**The fix is a participant-callable escape hatch:** after a wager
has been in `ACTIVE` for longer than `WAGER_RESOLUTION_TIMEOUT_MS`
— a window deliberately set well past any normal fight duration —
**either participant** can call `reclaim_stalled_wager`, which
refunds both stakes to their original depositors. No winner is
declared; this is the same pure-escrow-unwind shape as `settle_tie`.

This composes cleanly with the existing v5.1 safety net
(`cancel_expired_wager`, permissionless, also fires after a 10-minute
ACTIVE timeout with a 50/50 split). `reclaim_stalled_wager`
**preserves player rights independently** of whether the janitor
cron runs:

- `cancel_expired_wager` is the *permissionless backstop* — a bot,
  any wallet, anyone can call it; it splits the escrow 50/50; it
  exists so a stalled wager doesn't linger as visible state on the
  ladder forever.
- `reclaim_stalled_wager` is the *participant's explicit right* —
  only the two players in the wager can call it; it refunds each
  party their own original stake; it answers the audit-facing
  question "what stops the referee from holding my SUI hostage?"
  with "I do, on chain, after the timeout."

Both can co-exist. In practice the permissionless backstop will
usually fire first (10 min < 30 min). The participant-rights path
exists so the trustless guarantee is *named* and *exercisable*, not
implicit in a janitor function whose docstring says "anyone."

**Timeout sizing — the key safety property.** The timeout MUST be
clearly longer than any fight could legitimately take. The
constraint is two-sided:

- *Too short* → a losing player can call `reclaim_stalled_wager`
  mid-fight to escape a loss. Catastrophic — converts every wager
  into a free option for whoever was losing.
- *Too long* → players' SUI sits stranded if the referee really
  has failed; bad UX, audit-facing concern.

Concrete numbers: a typical fight in this game resolves in under 5
minutes (frontend turn-pause ceiling ≤ 60s × small turn count, plus
chain settle latency). The existing `SETTLEMENT_TIMEOUT_MS` (10
min, v5.1) is already comfortably past that. Recommended
`WAGER_RESOLUTION_TIMEOUT_MS = 30 min` — 6× the typical fight
duration, 3× the existing janitor timeout, room for chain
congestion and server-restart recovery. **Tunable** (constant lives
in `arena.move`); revisit after live data shows the 99th
percentile of settle latency.

The on-chain assertion is `clock::timestamp_ms(clock) >=
wager.accepted_at + WAGER_RESOLUTION_TIMEOUT_MS` — a player
calling early aborts with `EWagerNotStalled (= 19)`. There is no
"trust me, the fight is over" path; the only way to fire the
escape hatch is the clock.

## 4. Wager state machine

### States

| Status | Constant | Description |
|---|---|---|
| 0 | `STATUS_WAITING` | Creator has opened the wager; creator's stake in `escrow`; no challenger yet. **Same semantics as v5.1.** |
| 1 | `STATUS_ACTIVE` | Both stakes in `escrow`; fight is on. **Same semantics as v5.1.** |
| 2 | `STATUS_SETTLED` | Final terminal state. **Same semantics as v5.1.** |
| 3 | `STATUS_PENDING_APPROVAL` | **NEW.** Challenger has staked; awaiting creator approval. Challenger's stake in `challenger_escrow`, creator's stake still in `escrow`. |

Numbering preserves v5.1's `WAITING=0`, `ACTIVE=1`, `SETTLED=2` so any
downstream indexers that read the raw u8 don't have to renumber.

### Transitions

```
                             create_wager
                             (creator pays stake)
              ┌─────┐                              ┌────────────┐
              │ NEW │ ────────────────────────────▶│  WAITING   │
              └─────┘                              └────────────┘
                                                    │      ▲ │
                                                    │      │ │ cancel_wager
                                                    │      │ │ (creator)
                                                    │      │ ▼
                                                    │     ┌────────────┐
                                                    │     │  SETTLED   │
                                                    │     └────────────┘
                                                    │
                            request_accept_wager    │      (level-±1 check + stake match)
                            (challenger pays stake) ▼
                                  ┌──────────────────────────┐
                                  │   PENDING_APPROVAL       │
                                  │                          │
                                  │ creator's stake → escrow │
                                  │ challenger's stake →     │
                                  │   challenger_escrow      │
                                  │ pending_challenger set   │
                                  └──────────────────────────┘
                                    │           │              │
                  approve_challenger │ decline_  │  withdraw_   │  cancel_expired_
                  (creator)          │ challenger│  challenge   │  challenge
                                     │ (creator) │  (challenger)│  (anyone, t > CHALLENGE_TIMEOUT_MS)
                                     ▼           ▼              ▼
                            ┌──────────┐    ┌────────────────────────────────┐
                            │  ACTIVE  │    │  back to WAITING               │
                            └──────────┘    │  challenger_escrow refunded    │
                                  │         │  pending_challenger cleared    │
                                  │         └────────────────────────────────┘
                                  │
              settle_wager        │
              settle_tie          │
              admin_cancel_wager  │
              cancel_expired_     │
              wager (ACTIVE arm)  │
              reclaim_stalled_    │   ← v5.2 referee-liveness escape hatch
              wager               │     (either participant, t > WAGER_RESOLUTION_TIMEOUT_MS;
              (participant)       ▼      both stakes refunded, no winner)
                            ┌──────────┐
                            │  SETTLED │
                            └──────────┘
```

| From | Function | Caller | To | Funds movement |
|---|---|---|---|---|
| (none) | `create_wager` | creator | `WAITING` | creator's stake → `escrow` |
| `WAITING` | `cancel_wager` | creator | `SETTLED` | `escrow` → creator (refund) |
| `WAITING` | `request_accept_wager` | challenger (level-±1) | `PENDING_APPROVAL` | challenger's stake → `challenger_escrow` |
| `WAITING` | `admin_cancel_wager` | TREASURY | `SETTLED` | `escrow` → creator |
| `WAITING` | `cancel_expired_wager` | anyone (after `MATCH_EXPIRY_MS`) | `SETTLED` | `escrow` → creator |
| `PENDING_APPROVAL` | `approve_challenger` | creator | `ACTIVE` | `challenger_escrow` merges into `escrow` |
| `PENDING_APPROVAL` | `decline_challenger` | creator | `WAITING` | `challenger_escrow` → challenger |
| `PENDING_APPROVAL` | `withdraw_challenge` | **challenger** | `WAITING` | `challenger_escrow` → challenger |
| `PENDING_APPROVAL` | `cancel_expired_challenge` | anyone (after `CHALLENGE_TIMEOUT_MS`) | `WAITING` | `challenger_escrow` → challenger |
| `PENDING_APPROVAL` | `admin_cancel_wager` | TREASURY | `SETTLED` | both escrows refunded: `escrow` → creator, `challenger_escrow` → challenger |
| `ACTIVE` | `settle_wager` | TREASURY | `SETTLED` | 95% to winner, 5% TREASURY (unchanged from v5.1) |
| `ACTIVE` | `settle_tie` | TREASURY | `SETTLED` | refund both, no fee (unchanged from v5.1) |
| `ACTIVE` | `admin_cancel_wager` | TREASURY | `SETTLED` | 50/50 split (unchanged from v5.1) |
| `ACTIVE` | **`reclaim_stalled_wager`** *(v5.2 NEW)* | **either participant** (after `WAGER_RESOLUTION_TIMEOUT_MS`) | `SETTLED` | **refund both: `escrow` split into the two original stakes, each transferred to its original depositor. No platform fee. Emits `WagerReclaimed`. Closes the [referee-liveness gap](#35-referee-liveness--the-escape-hatch).** |
| `ACTIVE` | `cancel_expired_wager` | anyone (after `SETTLEMENT_TIMEOUT_MS`) | `SETTLED` | 50/50 split (unchanged from v5.1) |

## 5. Struct changes

### `WagerMatch`

```move
public struct WagerMatch has key {
    id: UID,
    player_a: address,
    /// Snapshot of the creator's character level at create time. The
    /// ±1 bracket check in request_accept_wager compares against this,
    /// not the live character level — so a creator who levels up
    /// while WAITING doesn't lock out their original bracket of
    /// possible challengers.
    player_a_level: u8,                              // NEW
    player_b: Option<address>,
    stake_amount: u64,
    escrow: Balance<SUI>,
    /// Challenger's stake while in STATUS_PENDING_APPROVAL. Empty
    /// (Balance::zero) in WAITING / ACTIVE / SETTLED — the funds
    /// either haven't arrived yet (WAITING) or have been merged into
    /// `escrow` on approval (ACTIVE / SETTLED).
    challenger_escrow: Balance<SUI>,                 // NEW
    /// Address of the wallet whose request is currently pending. Set
    /// by request_accept_wager, cleared by approve / decline /
    /// withdraw / cancel_expired_challenge.
    pending_challenger: Option<address>,             // NEW
    /// Timestamp the current pending_challenger was registered.
    /// Drives the CHALLENGE_TIMEOUT_MS expiry path.
    pending_at: u64,                                 // NEW
    status: u8,
    created_at: u64,
    accepted_at: u64,
    settled_at: u64,
}
```

### New constants

```move
// ===== Status constants =====
const STATUS_WAITING: u8 = 0;          // unchanged
const STATUS_ACTIVE: u8 = 1;           // unchanged
const STATUS_SETTLED: u8 = 2;          // unchanged
const STATUS_PENDING_APPROVAL: u8 = 3; // NEW

// ===== Timeouts =====
const MATCH_EXPIRY_MS: u64 = 600_000;        // unchanged (WAITING auto-refund)
const SETTLEMENT_TIMEOUT_MS: u64 = 600_000;  // unchanged (ACTIVE auto-split)
/// NEW — PENDING_APPROVAL auto-refund. Short enough that a creator who
/// abandons the tab doesn't lock the challenger out for long, long
/// enough for a real creator to see the notification, scout, and
/// click. Recommended starting value: 5 minutes. Tunable; revisit
/// after live data.
const CHALLENGE_TIMEOUT_MS: u64 = 300_000;   // NEW

/// NEW — Referee-liveness escape hatch. After a wager has been ACTIVE
/// for this long with no settlement, either participant can call
/// `reclaim_stalled_wager` to refund both stakes (no winner). MUST
/// be clearly longer than any legitimate fight could take — see
/// §3.5 for the safety derivation. Recommended starting value: 30
/// minutes (6× typical fight duration; 3× the v5.1 SETTLEMENT_
/// TIMEOUT_MS janitor). Tunable; tighten once the 99th-percentile
/// settle latency is measured on live testnet.
const WAGER_RESOLUTION_TIMEOUT_MS: u64 = 1_800_000;  // NEW

// ===== Level bracket =====
/// NEW — Inclusive bracket width. ±1 means accepter level must
/// satisfy |challenger_level - creator_level_snapshot| <= 1. Set as
/// a constant so a future v5.3 can widen the bracket without a Move
/// rewrite — only a const bump + republish.
const LEVEL_BRACKET: u8 = 1;                 // NEW
```

## 6. New / changed error constants

Existing v5.1 codes 0..11 stay put. v5.2 appends:

```move
/// v5.2 — Challenger's character level is outside the ±LEVEL_BRACKET
/// window from the creator's snapshot level. Raised by
/// request_accept_wager.
const ELevelOutOfBracket: u64 = 12;

/// v5.2 — A function that requires STATUS_PENDING_APPROVAL was called
/// on a wager in a different status (WAITING / ACTIVE / SETTLED).
/// Raised by approve_challenger, decline_challenger,
/// withdraw_challenge, cancel_expired_challenge.
const ENotPendingApproval: u64 = 13;

/// v5.2 — request_accept_wager was called on a wager that already has
/// a pending_challenger. Only one challenge in flight at a time.
const EChallengerSlotTaken: u64 = 14;

/// v5.2 — The wrong wallet called a creator-only PENDING_APPROVAL
/// transition (approve_challenger or decline_challenger). Distinct
/// from ENotPlayerA (= 4), which is reserved for WAITING-state
/// creator-only calls (cancel_wager), so error-code triage stays
/// unambiguous for indexers + clients.
const ENotCreatorForApproval: u64 = 15;

/// v5.2 — withdraw_challenge was called by a wallet that isn't the
/// pending_challenger.
const ENotPendingChallenger: u64 = 16;

/// v5.2 — cancel_expired_challenge was called before CHALLENGE_TIMEOUT_MS
/// has elapsed. Mirror of v5.1's ENotExpired but for the challenge
/// expiry path; kept as a distinct code so client copy can be specific.
const EChallengeNotExpired: u64 = 17;

/// v5.2 — reclaim_stalled_wager was called on a wager whose status is
/// not STATUS_ACTIVE. Kept distinct from the existing EMatchNotActive
/// (= 2, settle_wager's check) so frontend abort-humanizer copy can
/// be specific to the participant-rights escape-hatch path.
const ENotActiveForReclaim: u64 = 18;

/// v5.2 — reclaim_stalled_wager was called before
/// WAGER_RESOLUTION_TIMEOUT_MS has elapsed since wager.accepted_at.
/// The only "trust me the fight is over" defence the contract has;
/// player-facing copy reads e.g. "Wager isn't stalled yet —
/// settlement window still open."
const EWagerNotStalled: u64 = 19;

/// v5.2 — reclaim_stalled_wager was called by a wallet that isn't
/// one of the wager's two participants (player_a or player_b).
/// Distinct from ENotPlayerA (= 4) because the failure surface is
/// "not a participant" rather than the WAITING-state "not the
/// creator"; kept separate for client copy + indexer triage.
const ENotWagerParticipant: u64 = 20;
```

If `request_accept_wager` chooses to delegate to the
`EAlreadyHasOpenWager (= 11)` check for "challenger is already a
creator themselves" — see [§7.5](#75-the-already-has-open-wager-check)
— no new code is needed for that case.

## 7. New / changed functions

### 7.1 `create_wager` — minor change (snapshot creator level)

```move
public fun create_wager(
    stake: Coin<SUI>,
    creator_character: &Character,                  // NEW arg
    registry: &mut OpenWagerRegistry,
    clock: &Clock,
    ctx: &mut TxContext,
);
```

**New behavior vs v5.1:**

- Takes a `&Character` reference so it can read
  `character::level(creator_character)` and stamp `player_a_level` on
  the new `WagerMatch`. Asserts
  `character::owner(creator_character) == sender` to prevent
  level-spoofing via a borrowed Character.
- Asserts `character::is_fight_locked` is FALSE (defence — chain
  already covers the equip / unequip paths; this stops a creator
  opening a wager while another fight is mid-flight).
- Initializes `challenger_escrow: balance::zero()`,
  `pending_challenger: option::none()`, `pending_at: 0`.

**Everything else identical to v5.1.** Stake locks, registry insert,
`WagerCreated` event, share-object.

### 7.2 `request_accept_wager` — REPLACES `accept_wager`

```move
public fun request_accept_wager(
    wager: &mut WagerMatch,
    stake: Coin<SUI>,
    challenger_character: &Character,               // NEW arg
    registry: &OpenWagerRegistry,
    clock: &Clock,
    ctx: &mut TxContext,
);
```

**Assertions (in this order — each abort gives the user a more
specific error than the next):**

1. `wager.status == STATUS_WAITING` → `EMatchNotWaiting`
2. `option::is_none(&wager.pending_challenger)` → `EChallengerSlotTaken`
   *(belt-and-suspenders; status check above is the primary guard)*
3. Sender is not already a creator: `!table::contains(&registry.table, sender)` → `EAlreadyHasOpenWager` *(v5.1 rule, preserved)*
4. Sender is not the wager's creator: `sender != wager.player_a` → `ECannotJoinOwnMatch`
5. `character::owner(challenger_character) == sender` → `ENotOwner` *(prevent level-spoofing)*
6. Level bracket: `abs(character::level(challenger_character) - wager.player_a_level) <= LEVEL_BRACKET` → `ELevelOutOfBracket`. Implementation note: `u8` subtraction in Move underflows; compute as two-side compare, e.g.

   ```move
   let cl = character::level(challenger_character);
   let pl = wager.player_a_level;
   let in_bracket = if (cl >= pl) {
       cl - pl <= LEVEL_BRACKET
   } else {
       pl - cl <= LEVEL_BRACKET
   };
   assert!(in_bracket, ELevelOutOfBracket);
   ```

7. `coin::value(&stake) == wager.stake_amount` → `EStakeMismatch`

**Mutations:**

- `balance::join(&mut wager.challenger_escrow, coin::into_balance(stake))` — challenger's stake parks in the dedicated escrow.
- `wager.pending_challenger = option::some(sender)`
- `wager.pending_at = clock::timestamp_ms(clock)`
- `wager.status = STATUS_PENDING_APPROVAL`

**Event:**

```move
public struct ChallengeRequested has copy, drop {
    match_id: ID,
    player_a: address,            // creator (for indexer convenience)
    challenger: address,
    challenger_level: u8,         // for client display
    stake: u64,
    pending_at: u64,
}
```

### 7.3 `approve_challenger` — creator approves

```move
public fun approve_challenger(
    wager: &mut WagerMatch,
    clock: &Clock,
    ctx: &mut TxContext,
);
```

**Assertions:**

1. `wager.status == STATUS_PENDING_APPROVAL` → `ENotPendingApproval`
2. `tx_context::sender(ctx) == wager.player_a` → `ENotCreatorForApproval`
3. `option::is_some(&wager.pending_challenger)` → `ENoOpponent` *(safety; status invariant should already guarantee this)*

**Mutations:**

- Withdraw all of `challenger_escrow` and `balance::join` it into `escrow`.
- `wager.player_b = wager.pending_challenger`
- `wager.pending_challenger = option::none()`
- `wager.pending_at = 0`
- `wager.status = STATUS_ACTIVE`
- `wager.accepted_at = clock::timestamp_ms(clock)`

**Event:** Reuse the existing `WagerAccepted` shape so v5.1 indexers
keep working. (The intermediate `ChallengeRequested` event already
tells indexers about the new state machine.)

```move
event::emit(WagerAccepted {
    match_id,
    player_a: wager.player_a,
    player_b: <approved challenger>,
    total_stake: balance::value(&wager.escrow),
});
```

### 7.4 `decline_challenger` — creator rejects

```move
public fun decline_challenger(
    wager: &mut WagerMatch,
    ctx: &mut TxContext,
);
```

**Assertions:**

1. `wager.status == STATUS_PENDING_APPROVAL` → `ENotPendingApproval`
2. `tx_context::sender(ctx) == wager.player_a` → `ENotCreatorForApproval`
3. `option::is_some(&wager.pending_challenger)` → `ENoOpponent`

**Mutations:**

- Withdraw `challenger_escrow` to a `Coin<SUI>` and `transfer::public_transfer` it back to the challenger address stored in `pending_challenger`.
- `wager.pending_challenger = option::none()`
- `wager.pending_at = 0`
- `wager.status = STATUS_WAITING`

**Event:**

```move
public struct ChallengeDeclined has copy, drop {
    match_id: ID,
    player_a: address,
    challenger: address,
    refund: u64,
}
```

### 7.5 `withdraw_challenge` — **challenger** rescinds their own request

This is the strict-strictly-stronger-than-lock-on-approval lever.
A challenger never has to wait on the creator. They can always
pull their stake back themselves.

```move
public fun withdraw_challenge(
    wager: &mut WagerMatch,
    ctx: &mut TxContext,
);
```

**Assertions:**

1. `wager.status == STATUS_PENDING_APPROVAL` → `ENotPendingApproval`
2. `option::is_some(&wager.pending_challenger)` → `ENoOpponent`
3. `tx_context::sender(ctx) == *option::borrow(&wager.pending_challenger)` → `ENotPendingChallenger`

**Mutations:** Identical to `decline_challenger` — refund the
challenger, clear pending fields, status → `WAITING`.

**Event:**

```move
public struct ChallengeWithdrawn has copy, drop {
    match_id: ID,
    challenger: address,
    refund: u64,
}
```

### 7.5.bis The "already-has-open-wager" check

`request_accept_wager` keeps the v5.1 `EAlreadyHasOpenWager` guard so
a creator can't open a wager *and* simultaneously be pending on
someone else's wager. The dual-role case is allowed by the chain only
in this sequence: settle/cancel your own wager → request to accept
someone else's. Same invariant as v5.1; no new code, just enforce in
`request_accept_wager`.

### 7.6 `cancel_expired_challenge` — anyone, after timeout

```move
public fun cancel_expired_challenge(
    wager: &mut WagerMatch,
    clock: &Clock,
    ctx: &mut TxContext,
);
```

**Assertions:**

1. `wager.status == STATUS_PENDING_APPROVAL` → `ENotPendingApproval`
2. `clock::timestamp_ms(clock) >= wager.pending_at + CHALLENGE_TIMEOUT_MS` → `EChallengeNotExpired`
3. `option::is_some(&wager.pending_challenger)` → `ENoOpponent`

**Mutations:** Refund the challenger (same as `decline_challenger`),
clear pending fields, status → `WAITING`.

**Event:** Emit a `ChallengeExpired` (mirrors `WagerCancelled`'s role
for the WAITING-expiry path) so indexers can attribute the refund
correctly:

```move
public struct ChallengeExpired has copy, drop {
    match_id: ID,
    challenger: address,
    refund: u64,
}
```

### 7.7 `admin_cancel_wager` — extended for PENDING_APPROVAL

```move
public fun admin_cancel_wager(
    wager: &mut WagerMatch,
    registry: &mut OpenWagerRegistry,
    clock: &Clock,
    ctx: &mut TxContext,
);
```

Same TREASURY-only guard. v5.2 adds a third branch for
`STATUS_PENDING_APPROVAL`:

- Refund `escrow` (the creator's stake) to creator.
- Refund `challenger_escrow` (the challenger's stake) to challenger.
- Status → `SETTLED`; remove creator from registry; emit `WagerRefunded`
  with `refund_each = stake_amount` (each side gets back exactly their
  own stake since neither escrow was merged yet).

### 7.8 `cancel_expired_wager` — unchanged in WAITING/ACTIVE arms

v5.1's two branches stay. The PENDING_APPROVAL arm is handled by the
dedicated `cancel_expired_challenge` above (different timeout, different
event, and the "anyone can call" anti-strand role is the same).

### 7.9 `settle_wager`, `settle_tie` — unchanged

Both operate on `STATUS_ACTIVE` and have no dependency on the new
state. **No behavior change required.** They MUST refuse to act on
`STATUS_PENDING_APPROVAL` (already guaranteed by the existing
`wager.status == STATUS_ACTIVE` assertion → `EMatchNotActive`).

### 7.10 `cancel_wager` — unchanged

Still creator-only, still WAITING-only. The PENDING_APPROVAL state has
the challenger's stake in flight so a unilateral creator cancel from
that state would strand the challenger — instead the creator must
`decline_challenger` first (which returns to WAITING) and then
`cancel_wager`. Two-tx flow but no funds risk.

### 7.11 `reclaim_stalled_wager` — **either participant**, after `WAGER_RESOLUTION_TIMEOUT_MS`

The referee-liveness escape hatch from [§3.5](#35-referee-liveness--the-escape-hatch).
Either player in an `ACTIVE` wager can call this once the
settlement window has clearly elapsed, refunding both stakes with
no winner declared. Pure escrow unwind — same safety shape as
`settle_tie`, but invoked by a participant rather than TREASURY.

```move
public fun reclaim_stalled_wager(
    wager: &mut WagerMatch,
    registry: &mut OpenWagerRegistry,
    clock: &Clock,
    ctx: &mut TxContext,
);
```

**Assertions (in this order — each abort gives the user a more
specific error than the next):**

1. `wager.status == STATUS_ACTIVE` → `ENotActiveForReclaim`
   *(distinct code from settle_wager's `EMatchNotActive` so client
   abort-humanizer copy can be specific to the participant-rights
   path)*
2. `option::is_some(&wager.player_b)` → `ENoOpponent`
   *(safety; status invariant should already guarantee this)*
3. `clock::timestamp_ms(clock) >= wager.accepted_at + WAGER_RESOLUTION_TIMEOUT_MS`
   → `EWagerNotStalled` *(the timeout gate that prevents mid-fight
   abuse — see §3.5)*
4. `tx_context::sender(ctx) == wager.player_a || sender == *option::borrow(&wager.player_b)`
   → `ENotWagerParticipant`

**Mutations:**

- Split `escrow` exactly in half (both stakes were equal at
  `accept_wager` time per `EStakeMismatch`; `wager.stake_amount`
  is the canonical per-side value). Transfer `stake_amount` to
  `wager.player_a`; transfer the rest (handles any odd-MIST
  rounding) to the player_b address. Same shape as `settle_tie`.
- `wager.status = STATUS_SETTLED`
- `wager.settled_at = clock::timestamp_ms(clock)`
- Remove creator from `OpenWagerRegistry` if present.

**Why not `STATUS_REFUNDED` as a distinct terminal?** Considered
but rejected. `settle_tie` already terminates to `STATUS_SETTLED`
with the same pure-unwind shape; introducing a separate status u8
would break uniform terminal handling on existing v5.1 indexers
and add a value (`= 4`) that downstream callers must learn. The
**event** is what indexers should key on — see below.

**Event:**

```move
public struct WagerReclaimed has copy, drop {
    match_id: ID,
    player_a: address,
    player_b: address,
    refund_each: u64,
    /// Address of the participant who called reclaim. Useful for
    /// indexers tracking "which side gave up first" — a soft signal
    /// of which player believed the referee had failed.
    reclaimed_by: address,
    /// Cycle time the wager spent in ACTIVE before reclaim fired.
    /// Diagnostic; helps tune WAGER_RESOLUTION_TIMEOUT_MS post-launch.
    elapsed_ms: u64,
}
```

A `WagerReclaimed` event is **semantically distinct** from
`WagerRefunded` (admin-cancel split), `WagerTied` (mutual KO), and
`WagerCancelled` (creator-cancel from WAITING). Indexers should
treat the four as four separate terminal categories: settle, tie,
admin-refund, participant-reclaim.

**Relationship to `cancel_expired_wager`** *(v5.1, unchanged)*:
both can fire on `ACTIVE` past a timeout; both refund both sides
without declaring a winner. `cancel_expired_wager` is permissionless
(anyone, fires at `SETTLEMENT_TIMEOUT_MS = 10 min`, splits 50/50
via the existing v5.1 code path); `reclaim_stalled_wager` is
participant-only (fires at `WAGER_RESOLUTION_TIMEOUT_MS = 30 min`,
refunds each side their own stake). In practice the permissionless
backstop usually fires first; the participant path exists to make
the trustless guarantee *named* and *exercisable*. Both coexist;
neither blocks the other.

### 7.12 `accept_wager` — **REMOVED**

v5.2 deletes the v5.1 `accept_wager` entry. The replacement is
`request_accept_wager` + `approve_challenger`. v5.1 callers (frontend,
server) will need to be updated to the new two-step flow. Because v5.2
is a fresh package publish (see [Dependencies](#dependencies)), v5.1
remains callable at its own package id for any wager whose lifecycle
straddles the cutover — they settle under v5.1 rules; new wagers go
under v5.2.

## 8. Escrow timing & failure-mode table

Every escrow transition. The invariant we're proving: **no path can
strand or burn SUI.**

| State entered | Funds in `escrow` | Funds in `challenger_escrow` | What can move funds out, and to whom |
|---|---|---|---|
| `WAITING` | creator's stake | empty | `cancel_wager` (creator) / `admin_cancel_wager` (TREASURY) / `cancel_expired_wager` (anyone after 10 min) → all refund creator |
| `PENDING_APPROVAL` | creator's stake | challenger's stake | `approve_challenger` (creator) → merges into `escrow`, fight starts. `decline_challenger` (creator) → refunds challenger. `withdraw_challenge` (**challenger**) → refunds challenger. `cancel_expired_challenge` (**anyone** after `CHALLENGE_TIMEOUT_MS = 5 min`) → refunds challenger. `admin_cancel_wager` (TREASURY) → refunds both. |
| `ACTIVE` | both stakes | empty | `settle_wager` (TREASURY) / `settle_tie` (TREASURY) / `admin_cancel_wager` (TREASURY) / `cancel_expired_wager` (anyone after `SETTLEMENT_TIMEOUT_MS = 10 min`, 50/50 split, v5.1 unchanged) / **`reclaim_stalled_wager` (either participant after `WAGER_RESOLUTION_TIMEOUT_MS = 30 min`, refund-each, v5.2 NEW — see [§3.5](#35-referee-liveness--the-escape-hatch))** |
| `SETTLED` | empty | empty | terminal — no further movement |

### Failure modes — explicit

1. **Creator abandons in PENDING_APPROVAL.** Challenger has three recovery paths:
   - Self-service: `withdraw_challenge` (immediate, challenger pays gas).
   - Permissionless: `cancel_expired_challenge` after 5 minutes (anyone — typically a server cron, but any wallet can clear the wager).
   - Adversarial: TREASURY `admin_cancel_wager` (full refund both sides).
2. **Challenger abandons in PENDING_APPROVAL.** Creator has two paths:
   - `decline_challenger` (immediate, restores `WAITING`).
   - Wait `CHALLENGE_TIMEOUT_MS`, let `cancel_expired_challenge` clean up automatically.
3. **Both abandon.** `cancel_expired_challenge` runs after timeout; refunds challenger, returns to `WAITING`; then either `cancel_expired_wager` (after the longer 10-min `MATCH_EXPIRY_MS`) refunds creator. **Total max strand: `CHALLENGE_TIMEOUT_MS + MATCH_EXPIRY_MS = 15 minutes`, with the challenger able to exit at any time during the first 5 min.**
4. **Server crashes between request and approval.** Server is not a participant in either request_accept_wager or approve_challenger / decline_challenger transitions. No effect on funds.
5. **Referee (server / TREASURY) never reports outcome on an `ACTIVE` wager.** *The audit-flagged single-point-of-failure case.* Three recovery paths, in order of timing:
   - `t = 0..10 min`: server is expected to settle in the fast path. No external action needed.
   - `t > 10 min`: `cancel_expired_wager` becomes callable by **anyone** (v5.1 permissionless janitor, 50/50 split). A monitoring bot or any wallet can clear the wager. Players get their stake back via the even-split path.
   - `t > 30 min`: **`reclaim_stalled_wager` becomes callable by either participant** (v5.2 NEW, [§7.11](#711-reclaim_stalled_wager--either-participant-after-wager_resolution_timeout_ms)). Refunds each side their original stake (not a split). This is the explicit player-rights path — exists even if the permissionless backstop is somehow blocked, and exists so the trustless guarantee is named and exercisable on chain.

   **Key invariant: `WAGER_RESOLUTION_TIMEOUT_MS` is set well past any legitimate fight duration.** Mid-fight abuse (a losing player calling reclaim to escape) is blocked by the clock gate — `EWagerNotStalled` aborts any call before the timeout. Typical fight ≤ 5 min; reclaim available at 30 min; 25-min buffer.
6. **TREASURY-key compromise — fund-direction case.** If TREASURY's key is stolen, the attacker could call `settle_wager` to direct payout to an attacker-controlled winner address. This specific class is addressed by `settle_wager_attested` (separately tracked in v5.2 backlog — requires dual signatures from both participants on the outcome). The reclaim path above does NOT defend against this case (attacker can settle before the 30-min window opens); it defends against the *liveness* case (no settle at all).

## 11. Dependencies

1. **v5.1 must merge to `main` first.** Standing rule (see
   `MAINNET_PREP.md`): no merge to `main` until v5.1 QA fully done
   AND the external smart-contract audit clears v5.2. v5.2 is cut as
   a clean milestone branch off `main`, not off
   `feature/v5.1-contracts`. This keeps the v5.2 audit scope free of
   any "what's the diff vs the not-yet-merged v5.1" ambiguity.

2. **v5.2 is a fresh `sui client publish`.** Cannot patch the v5.1
   package in place (chain-side `WagerMatch` struct adds 4 fields;
   adding fields to a published struct is impossible on Sui — UpgradeCap
   permits only backward-compatible additions to function bodies and
   new entries, not struct shape changes). Same constraint that forced
   v5.0 → v5.1 to be a fresh publish.

3. **Bundle with the other v5.2 contract work.** The v5.2 publish
   should ship the wager-fairness changes together with
   `settle_wager_attested`, `sui::random`-based loot/RNG, and
   `respec_character`. One publish, one audit pass, one cut-over
   protocol. Splitting these into separate publishes triples the
   audit cost and triples the cut-over risk.

4. **External smart-contract audit.** Required before any mainnet
   publish. v5.2 testnet → audit → audit fixes → audit re-check →
   mainnet. Roughly 2-4 weeks audit lead time, separate from
   implementation time.

5. **No new external chain dependencies** (no other modules need to
   change). `arena.move` already imports
   `sui_combats::character::Character`-style references where needed
   in v5.1; the new `&Character` args to `create_wager` /
   `request_accept_wager` only require the existing `character`
   module's read accessors.

## 12. Open questions

> **Summary — what this spec closes.** v5.2 fairness shuts the
> high-level-farms-low-level abuse (level ±1 bracket), the
> any-wallet-can-snipe abuse (creator approval), AND the
> centralized-referee single-point-of-failure / funds-in-limbo gap
> flagged by external review (`reclaim_stalled_wager` participant
> escape hatch — see [§3.5](#35-referee-liveness--the-escape-hatch),
> [§7.11](#711-reclaim_stalled_wager--either-participant-after-wager_resolution_timeout_ms)).
> What it does NOT close: the TREASURY-can-pick-the-wrong-winner trust
> assumption — that's `settle_wager_attested`, still in the v5.2
> backlog as separate work.

0. **`WAGER_RESOLUTION_TIMEOUT_MS` — exact value.** 30 min is the
   starting recommendation (6× typical fight, 3× the existing
   `SETTLEMENT_TIMEOUT_MS = 10 min` janitor, room for chain
   congestion). Could be tightened to 15 min once live data shows
   the 99th-percentile settle latency on testnet. **Must NOT drop
   below the longest legitimate fight duration** — that's the
   non-negotiable safety floor.
1. **`CHALLENGE_TIMEOUT_MS` — exact value.** 5 min is the starting
   recommendation. Could be 2 min (faster turnover, more strict on
   creator attention) or 10 min (more permissive). Decide pre-audit
   from live data; tunable post-mainnet by republish.
2. **`LEVEL_BRACKET` — ±1 vs ±2.** ±1 is the spec'd value. Question
   for the user: in the live Lv1-vs-Lv2 data the bracket effectively
   excludes a lot of legitimate matches in the early game when the
   active player base is thin. Could ±2 be more practical for testnet
   while keeping ±1 as the mainnet target? Easy to tune via the
   constant.
3. **Can the creator change their level-snapshot after PENDING_APPROVAL
   times out?** Currently the snapshot is fixed at create_wager. If
   the wager sits in WAITING for an hour and the creator levels up
   3× in between, the snapshot stays stale. Recommend keeping the
   snapshot fixed — the creator can `cancel_wager` and re-create if
   they want the new snapshot. Cheaper than adding a "refresh level
   snapshot" entry.
4. **What does `decline_challenger` look like in UX?** Should the
   creator be required to write a reason? (No — keep on-chain
   payload minimal; client can volunteer a chat message separately
   via the existing Tavern DM channel.)
5. **Multiple pending challengers** (queue rather than slot)? **Out
   of scope.** v5.2 ships single-slot PENDING_APPROVAL; if testnet
   shows demand for a queue, a v5.3 design adds a
   `pending_queue: vector<address>`. Defer until we have data.

## 14. Implementation deviations (2026-05-30)

Tracked here so future audits can diff spec-as-written vs
implementation-as-shipped. Each item is a judgment call made during
the Move translation pass; none change the user-facing semantics
defined in §3 / §4 / §7.

### 14.1 Three new abort codes beyond the 12–20 range

The spec §6 lists abort codes 12–20. Three assertions surfaced
during implementation that the spec named narratively without
assigning arena-side codes:

- **`ECreatorFightLocked: u64 = 21`** — `create_wager` when the
  creator's Character is currently fight-locked (defence-in-depth on
  top of the equipment module's fight-lock). Spec §7.1 mentions the
  assertion but doesn't name a code; 21 keeps clean separation from
  the v5.2 fairness range 12–20.
- **`ENotCharacterOwner: u64 = 22`** — `create_wager` /
  `request_accept_wager` when the passed Character isn't owned by
  the sender (anti-spoofing). Spec §7.1 / §7.2 refer to this as
  `ENotOwner`, but `character::ENotOwner = 5` would collide with
  `arena::EInvalidWinner = 5`. Arena's namespace needed its own
  non-colliding code.
- **`EWrongExpiryEntrypoint: u64 = 23`** — `cancel_expired_wager`
  when status is PENDING_APPROVAL. Explicitly routes the caller to
  `cancel_expired_challenge` instead of falling through to a
  misleading ENoOpponent. Added 2026-05-30 to tie off the last

### 14.2 `EChallengerSlotTaken (= 14)` is unreachable via the public API

With the assertion order in §7.2 (status check before slot check),
code 14 only fires if `wager.status == STATUS_WAITING` AND
`pending_challenger == Some(_)` — a state the public API can't
produce, because `request_accept_wager` always moves
`WAITING → PENDING_APPROVAL` in the same tx that sets
`pending_challenger`. The assertion stays in `arena.move` as
belt-and-suspenders defence-in-depth. A test-only helper
`arena::force_status_for_testing(wager, status)` lets the test suite
forge the impossible state to verify the assertion is wired
correctly. Normal users hitting the same scenario abort with
`EMatchNotWaiting = 1` (the realistic failure surface, also tested).

### 14.3 `ECannotJoinOwnMatch (= 7)` is unreachable in practice

Same shape as v5.1: a creator trying to `request_accept_wager` on
their own wager hits `EAlreadyHasOpenWager = 11` first (the
registry check), because the creator is in the registry. Code 7
remains in the assertion order per spec §7.2 as belt-and-suspenders
for a theoretical creator-not-in-registry case. No test for code 7
directly; the user-facing case is covered by
`test_creator_request_own_aborts_already_open` (fires code 11).

### 14.4 `cancel_expired_wager` on PENDING_APPROVAL — RESOLVED 2026-05-30

The earlier revision of this spec documented a deviation where
`cancel_expired_wager` had no explicit PENDING_APPROVAL branch and
callers fell through to a misleading `ENoOpponent = 10`. Resolved
in commit hardening this deviation: `cancel_expired_wager` now
asserts `status != STATUS_PENDING_APPROVAL` early and aborts with
the dedicated `EWrongExpiryEntrypoint = 23` (see §14.1). Frontend
abort-humanizer gets a clear "use cancel_expired_challenge"
message; no funds risk. Direct triggering test:
`test_cancel_expired_wager_on_pending_approval_aborts`.

### 14.5 Test-only helper added to `character.move`

`character::set_level_for_testing(character: &mut Character, level: u8)`
— `#[test_only]`, sole purpose to bump a Character's level without
grinding XP through `update_after_fight` (which would also dirty
`wins` / `losses` and confuse fairness-test assertions). Minimal,
additive, does NOT compile into production builds. Necessary
scaffolding for the §10 level-bracket test group; no production
behaviour change.
