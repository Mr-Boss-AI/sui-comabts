# v5.2 Bug Ledger

Live-QA bugs surfaced during the v5.2 testnet cut-over and their fixes.
Each entry records: when, where, root cause, what the chain did, the
two-part fix, and what test pins it forever.

## 2026-05-31 — server-vs-chain lobby drift after restart + thin SDK envelope hid the abort code

**Surfaced by:** Live QA, immediately after the notification-modal
commit (`203ddcf`). Sx tried to open a fresh 0.1-SUI wager and got
`"create_wager aborted on-chain (see console for raw result)"` in the
bottom-corner toast plus a "1 Issue" badge. Both lobby boards showed
"No open wagers", so the UI offered no Cancel button.
**Severity:** Public-testnet blocker — a player could be locked out of
the wager flow indefinitely until they (a) figured out they had to call
`cancel_wager` directly on chain via the SDK, or (b) waited 10 min for
`cancel_expired_wager` to be callable by anyone. No way to recover from
the UI.
**Impact on chain funds:** None during the incident itself (escrow was
intact in the stranded WAITING wager). Treasury-signed
`admin_cancel_wager` refunded Sx's 0.1 SUI in full.

### Symptom

Two compounding failures stacked into one unactionable toast:

1. **Chain ↔ server drift.** Earlier in the day Sx had opened a 0.1-SUI
   wager (`0x44018b2f…d6ef`) that nobody accepted. When the server was
   restarted after the modal commit, the in-memory `wagerLobby` map was
   wiped. The on-chain `OpenWagerRegistry` still pinned Sx's wallet to
   that wager id (the chain has no idea the server restarted), so any
   new `create_wager` aborted with `EAlreadyHasOpenWager (11)`.
2. **Thin SDK envelope hid the code.** The frontend's
   `wager-preflight.ts::simulateWagerTx` ran `client.simulateTransaction`
   and dutifully passed the result to `assertTxSucceeded`. The 2.16
   `FailedTransaction.status.error` is an OBJECT (`ExecutionError` with
   a nested `MoveAbort.abortCode`), not a string. `assertTxSucceeded`
   only knew the string shape, fell through to the empty fallback, and
   the user saw `aborted on-chain (see console for raw result)` —
   useless. The actual abort code (11) was present in the envelope but
   never extracted, so the abort-code map was never consulted.

### Chain trace

| # | Object | Outcome |
|---|---|---|
| 1 | `OpenWagerRegistry` `0xabf10378…086e16` | `table.size = 1`; sole entry maps Sx → wager `0x44018b2f…d6ef` |
| 2 | `WagerMatch` `0x44018b2f…d6ef` | `status: 0` WAITING, `player_a` Sx, `escrow: 100_000_000 MIST`, no `player_b`, no `pending_challenger` — alive since `created_at: 1780218356014` |
| 3 | Treasury `admin_cancel_wager` `AZMEPrrZ5Arcy3Sh661CpRLeYN7q9BNPwRksx5wBZWT` | SUCCESS — refunded 0.1 SUI to Sx, status → SETTLED, registry table.size → 0 |
| 4 | Sx balance | 761_822_948 MIST → 861_822_948 MIST (Δ +100_000_000 = +0.1 SUI) ✓ |

### Root cause

Two independent bugs that only become visible together:

- **Boot rehydration was missing.** The server has been carrying the
  lobby purely in-memory since v5.1. `wagerLobby` is rebuilt only by
  fresh `create_wager` events arriving over WS — there was never a path
  from the on-chain registry back into the in-memory map after a restart.
  Players with WAITING/PENDING wagers across a restart lost UI access
  to them entirely.
- **`assertTxSucceeded` is string-only.** The 2.16 SDK serialises Move
  aborts as `{ $kind: 'MoveAbort', MoveAbort: { abortCode, location } }`
  inside `status.error`. Pre-fix the function only looked for
  `typeof error === 'string'`, so the structured envelope's abort code
  was never extracted. Defensive string fallbacks (`status.error`,
  `failed.error`, `failed.errorMessage`, `failed.cause`) all returned
  empty against the 2.16 shape; the generic "aborted on-chain" toast
  was all the user ever saw.

### Three-part fix

#### 1. One-shot rescue — `scripts/admin-cancel-stranded-wager.ts`

Treasury-signed `admin_cancel_wager` against the stranded id. Reads pre-
and post-cancel state (status, escrow, player_a balance, registry table
size) so the operator sees the rescue land cleanly. Re-uses the server's
own `adminCancelWagerOnChain` helper so we inherit the retry envelope +
registry-arg ordering and don't drift from production behaviour. Used
this exact path to unstick Sx on 2026-05-31.

#### 2. Boot rehydration — `server/src/data/lobby-rehydration.ts`

New module wired into `server/src/index.ts` boot. After
`sweepOrphanActiveWagers()` runs, `bootRehydrateLobby()` reads the
`OpenWagerRegistry` table dynamic fields, fetches each `WagerMatch`,
and re-adopts every WAITING + PENDING_APPROVAL entry into the in-memory
lobby via the existing `adoptWagerIntoLobby` helper. Status branches:

- **WAITING (0)** → adopt with character data (Supabase → DB lookup) or
  placeholder if no row (wallet-prefix name + `player_a_level` snapshot
  + zero stats). Creator sees their card with a Cancel button.
- **PENDING_APPROVAL (3)** → adopt + carry `pendingChallenger`
  (placeholder name; next WS auth refreshes).
- **ACTIVE (1)** → SKIP; `sweepOrphanActiveWagers` owns this row via
  the `wager_in_flight` Supabase table.
- **SETTLED (2)** → SKIP defensively + log. The v5.2 contract removes
  registry entries on settle, so a SETTLED row in the registry is a
  contract bug, not a recovery case.

DI-injected (`RehydrateDeps`) for testability — production wires real
chain + Supabase + lobby; the gauntlet drives every branch with mocks.

#### 3. Structured-abort humanizer — `frontend/src/lib/tx-result.ts`

Two new exports: `readStructuredAbort(err)` extracts code + module +
function + instruction from the 2.16 `ExecutionError` envelope, and
`formatStructuredAbort(abort, codes)` produces the same wire-shape
string `humanizeChainError` does for the regex path. `assertTxSucceeded`
now tries the structured path FIRST in every branch (FailedTransaction,
Transaction-with-failure, legacy fallback) before falling through to
the string fallbacks. The pre-existing string path is preserved
verbatim as defence for legacy wallets / pre-2.16 envelopes.

Also deepened the preflight diagnostic in `wager-preflight.ts` to log
`failedErrType`, `failedErr$kind`, `hasMoveAbort`, and `abortCode` so
the next SDK-shape drift is one log line away from a diagnosis instead
of a forensic exercise.

### What the user sees post-fix

Sx repeats the exact same scenario:
- Old toast: `"create_wager aborted on-chain (see console for raw
  result)"` (bottom corner, 5s fade).
- New toast: `"create_wager failed: You already have an open wager.
  Settle or cancel it before starting another. (at arena::create_wager:14)"`
  And — because the lobby rehydration ran at boot — the Cancel button
  is right there on the original wager card.

### Tests that pin this forever

- `scripts/qa-lobby-rehydration.ts` (NEW, 61/61) — drives every
  RehydrateDeps branch with mocks: WAITING + PENDING_APPROVAL adoption
  (with and without DB character), ACTIVE skip, SETTLED skip, unknown
  status as error, null fields as error, registry-read throw, per-wager
  throw, character-load throw (non-fatal placeholder), adoption
  rejection (idempotent), empty registry.
- `scripts/qa-arena-aborts.ts` (extended, 67 → 106) — new tests for the
  structured-abort path: `readStructuredAbort` correctness across
  MoveAbort + non-MoveAbort kinds; `formatStructuredAbort` matches the
  regex-path wire shape; `assertTxSucceeded` with the empty-`.message`
  2.16 envelope reaches the EAlreadyHasOpenWager copy (the literal bug
  case); every arena abort code round-trips through the structured
  path; legacy string envelope still humanizes correctly (no regression
  on pre-2.16 wallet shapes).

## 2026-05-30 — `decideAcceptOutcome` false-positive on creator-approve

**Surfaced by:** First live QA pass through the v5.2 wager flow.
**Severity:** Hard regression — no v5.2 wager fight could start.
**Impact on chain funds:** None. The auto-rollback's `admin_cancel_wager`
50/50-refunded both depositors cleanly.

### Symptom

Sx (creator) and Mr_Boss (challenger) ran the v5.2 handshake to the end:

1. Sx signed `create_wager` ✓
2. Mr_Boss signed `request_accept_wager` → wager → PENDING_APPROVAL ✓
3. Server populated `pendingChallenger`, broadcast `wager_lobby_updated` ✓
4. Sx's UI showed Approve / Decline correctly ✓
5. Sx clicked **Approve** → signed `approve_challenger` ✓
6. Chain went ACTIVE ✓
7. Frontend sent `wager_accepted` to server
8. **Server auto-rolled back the wager** instead of starting the fight.

Creator's toast: `"Auto-rolled back — stakes refunded. You had your own
open wager; the chain accept slipped past the client gate."`

### Chain trace

| # | Tx | Outcome |
|---|---|---|
| 1 | `approve_challenger` (Sx) — `6YbYMEPMFkfJQVaoMFUM5kHtsy8PrrWCJvoSg1CTDKp8` | **SUCCESS** — status PENDING_APPROVAL → ACTIVE, escrow merged to 0.2 SUI |
| 2 | `admin_cancel_wager` (TREASURY, server autoRollback "target") — `dpabzZ9utEqUJQUPJG8sPjkcsSs96SkpiisW9W3D7re` | **SUCCESS** — wager → SETTLED, 0.1 SUI back to each side |
| 3 | `admin_cancel_wager` (TREASURY, autoRollback "callerOwn" — same id) — `HTbaGbuyYyQTetwER4y1cAJK1WJB21Yxr5uEpw1rZMWa` | MoveAbort 6 `EMatchAlreadySettled` |
| 4 | retry 2/3 — `G9XNmkf8GB9VL5Q1gUzGzKQkB1kVoxykVno1k5iYQyBn` | MoveAbort 6 |
| 5 | retry 3/3 — `5ajNKbCrwhNyWSqkvrzUgxJPaNB7S1cCHer9YUHrAL7Q` | MoveAbort 6 |

**Chain was 100% correct.** `approve_challenger` does NOT check
`EAlreadyHasOpenWager (11)` — its only asserts are status (13), creator
(15), and pending challenger exists (10). The bug was server-side TS.

### Root cause

`server/src/ws/wager-accept-gate.ts::decideAcceptOutcome` step 3 was
written for the v5.1 silent-accept invariant:

> *"If the caller has their own open wager AND the chain status flipped
> to ACTIVE, that's the bug where the client gate failed to block a
> silent-accept — auto-rollback both wagers."*

In v5.2 the creator IS the caller of `wager_accepted` (after signing
`approve_challenger`), and the creator's "own wager" IS the target —
not a separate wager. The original check misclassified the legitimate
v5.2 approve flow as a silent-accept and auto-rolled back the wager
that should have just started a fight.

Pre-fix logic in step 3:
```typescript
const hasOwnWager = !!args.callerOwnWagerInLobby;
if (hasOwnWager || callerInQueue) {
  if (args.targetChainStatus === STATUS_ACTIVE) {
    return { kind: 'autoRollback', ... };
  }
}
```

There was no test for `ownWagerInLobby.wagerMatchId === targetWagerId`,
so v5.2 creator-approve fell into the busy/autoRollback branch.

### Related second bug (caught while fixing)

The post-gate proceed path in `handleWagerAccepted` resolved
`player_b = client.walletAddress`. In v5.1 that's correct (challenger
signed `accept_wager`). In v5.2 the caller is the creator (player_a),
so `player_b` MUST come from the lobby entry's `pendingChallenger.wallet`
(server-cached when the earlier `wager_request_accepted` handshake
arrived) — not from the caller wallet. Even with the gate fix, a v5.2
fight would have started with the wrong player_b otherwise.

### Two-part fix

#### 1. Gate guard — `wager-accept-gate.ts::decideAcceptOutcome`

```typescript
const ownWagerIsTarget =
  !!args.callerOwnWagerInLobby &&
  args.callerOwnWagerInLobby.wagerMatchId === args.targetWagerId;
const hasDifferentOwnWager = hasOwnWager && !ownWagerIsTarget;

if (hasDifferentOwnWager || callerInQueue) {
  // ... existing autoRollback / reject logic
}
```

When `ownWagerIsTarget`, step 3 is skipped entirely; flow falls through
to step 5 (proceed). The genuine silent-accept guard remains intact for
the case it was meant for — a caller with a DIFFERENT open wager.

#### 2. Player_b resolution — `wager-accept-gate.ts::resolveChallengerWallet`

New pure helper centralises the dispatch:

```typescript
export function resolveChallengerWallet(args: {
  callerWallet: string;
  creatorWallet: string;
  pendingChallengerWallet?: string;
}): ResolveChallengerOutcome
```

- `callerWallet != creatorWallet` → v5.1-legacy flow, challenger = caller.
- `callerWallet == creatorWallet AND pendingChallenger present` →
  v5.2-approve flow, challenger = pendingChallenger.wallet.
- `callerWallet == creatorWallet AND no pending` → defensive error
  (only happens if server restarted between `wager_request_accepted`
  and `wager_accepted`).

`handleWagerAccepted`'s proceed path now uses the resolved
`challengerWallet` for `dbInsertWagerInFlight.player_b`, the
`getCharacterByWallet` lookup for charB, AND the straggler-sweep loop
(the loop was using `client.walletAddress` and would miss the
challenger's stragglers in v5.2 because the caller is the creator).

### Tests pinning the fix

`scripts/qa-wager-accept-gate.ts` extended with 7 new assertions across
3 new test functions (total: 85/85 PASS, was 71/71):

| Test | Pins |
|---|---|
| `testV52CreatorApprovesOwnTarget` | own == target + ACTIVE → PROCEED (not autoRollback) |
| `testV52CallerHasDifferentOwnWagerStillRollsBack` | own != target + ACTIVE → autoRollback (silent-accept guard preserved) |
| `testV52CreatorApprovesOwnTargetPendingApprovalStaysClean` | own == target + PENDING_APPROVAL (probe lag) → reject, never rollback |
| `testV52ResolveChallengerV52ApproveFlow` | caller == creator + pending → challenger from pendingChallenger |
| `testV52ResolveChallengerV51LegacyFlow` | caller != creator → challenger = caller (v5.1 backward compat) |
| `testV52ResolveChallengerMissingPending` | caller == creator + no pending → clear error (server-restart edge) |
| `testV52ResolveChallengerCaseInsensitive` | wallet comparison is case-insensitive |

### What this means for QA

After the fix:
- Sx → create_wager → request from Mr_Boss → Sx approves → **fight starts**.
- Sx's stale wager `0x97c33887…064ea6` is already SETTLED on chain (from
  the autoRollback's `dpabzZ9u…`). Both wallets +0.1 SUI back. No
  manual cleanup needed.
- Re-run from a fresh create_wager.

## 2026-05-30 — StatAllocateModal shows "Remaining: 0" after level-up

**Surfaced by:** Same live-QA pass that exercised the
`decideAcceptOutcome` fix above. Mr_Boss won the wager fight, leveled
Lv1 → Lv2, payout correct, no rollback (the gate fix works). But the
celebration modal's "Allocate Stat Points" CTA opened the
`StatAllocateModal` showing **Remaining: 0** instead of the +3 the
GDD promises.

**Severity:** Soft — chain has the points, only the UI was hiding
them. No funds at risk. User-blocking for stat allocation until
the next session.

**Impact on chain funds:** None. The points were granted on chain
correctly; only the read path was broken.

### Diagnosis

The contract was correct. Read `Character` object
`0x620d5008…b4e42` directly:

```json
{
  "level": 2,
  "xp": "100",
  "unallocated_points": 3,    ← chain HAS the +3
  "wins": 1, "losses": 0, ...
}
```

`LevelUp` event also confirms via Sui GraphQL:
`{ character_id: …b4e42, owner: 0xf669789c…, new_level: 2, xp: 100, unallocated_points: 3 }`.

The Move code in `character.move::update_after_fight` lines 337-339
correctly does `character.unallocated_points += POINTS_PER_LEVEL`
(POINTS_PER_LEVEL = 3) inside the level-up loop. v5.2 didn't touch
`character.move`; this is the same code that worked in v5.1.

So Hypothesis A (data path drops the field), not Hypothesis B
(server in-memory not synced). Specifically, **three independent
drop points** along the chain → modal pipeline:

| # | Drop point | Effect |
|---|---|---|
| 1 | `fight-room.ts:691` — server only sends `character_updated_onchain` (chain-refetch trigger) after the post-fight chain update; no fresh `character_data`. | Frontend's `state.character.unallocatedPoints` stayed at 0 indefinitely (only `state.onChainCharacter.unallocatedPoints` got refreshed to 3 via the chain refetch). |
| 2 | `index.ts:63-78` — `/api/character` REST serializer omitted the field entirely. | Any caller of REST got an incomplete payload (the user explicitly flagged this). |
| 3 | `stat-points.ts:25-37` — `effectiveUnallocatedPoints` returned `min(server, chain)`. With server=0 and chain=3, `min(0, 3) = 0` → modal showed 0 even though chain would accept 3. | The symmetric race to v5.1's BUG 1: pre-fix only handled "server ahead, chain behind" — post-fix handles both directions correctly by trusting chain. |

Note: drop #1 makes drop #3 manifest. Even with #1 fixed (server
pushes character_data so server-mirror catches up), the user could
still race the modal open vs. the WS push — so drop #3 must be
fixed at the helper layer too. Defence in depth.

### Hypothesis B was disproved

Hypothesis B (server in-memory not re-synced after the chain retry
succeeded) was already correctly handled at
`fight-room.ts:686-689`:

```typescript
if (effects.leveledUp) {
  winnerCharRef.level = effects.newLevel;
  winnerCharRef.unallocatedPoints = effects.newUnallocatedPoints;
}
updateCharacter(winnerCharRef);
```

`updateCharacterOnChain` returns `effects.newUnallocatedPoints`
mirrored from the chain post-tx, so the server's in-memory record
DOES converge to chain truth. The gap was the absent WS push to the
client, not the server-side re-sync.

### Three-part fix

#### 1. `frontend/src/lib/stat-points.ts::effectiveUnallocatedPoints`

```typescript
if (chainPoints == null) return sanitize(serverPoints);
return sanitize(chainPoints);  // was: Math.min(s, c)
```

Chain is the binding gate for `allocate_points` (the Move assert
`total <= character.unallocated_points` is the authority). When chain
data is available, trust it. Handles both:
- Race A (chain=0, server=3 → return 0, no doomed tx)
- Race B (chain=3, server=0 → return 3, correct UX)

`isAwaitingChainCatchup` left asymmetric — only flags Race A (the
"catching up" banner only matters when server > chain, which is the
doomed-tx case).

#### 2. `server/src/utils/wire-sanitize.ts` — extract `sanitizeCharacter`

Moved out of `ws/handler.ts` to break the
`handler.ts → fight-room.ts → handler.ts` import cycle. Behaviour
identical (1-for-1 with the prior in-handler version including the
non-zero diagnostic log and `onChainObjectId` carry-through).

#### 3. `server/src/ws/fight-room.ts` — push fresh character_data after chain sync

```typescript
updateCharacter(winnerCharRef);
sendToWallet(winnerCharRef.walletAddress, {
  type: 'character_data',
  character: sanitizeCharacter(winnerCharRef),  // NEW
});
sendToWallet(winnerCharRef.walletAddress, { type: 'character_updated_onchain' });
```

Same push on the loser branch. The push fires AFTER `winnerCharRef`
has been mirrored from chain effects, so the wire payload reflects
chain truth.

#### 4. `server/src/index.ts` — REST `/api/character` serializer

Added `unallocatedPoints: character.unallocatedPoints` to the
response shape. Brings REST in line with WS.

### Live verification

After server restart with all four changes:

```
$ curl -s http://localhost:3001/api/character/0xf669789c…0590f33 | jq '.character | {level, xp, unallocatedPoints}'
{
  "level": 2,
  "xp": 100,
  "unallocatedPoints": 3    ← was missing pre-fix
}
```

Chain truth matches served payload. The frontend modal will now show
`Remaining: 3` on Mr_Boss's next reconnect / page refresh.

### Tests pinning the fix

`scripts/qa-stat-points.ts` extended: 50/50 pass (was 39/39). New
sections:

| Section | Pins |
|---|---|
| [3] Race B coverage | `effectiveUnallocatedPoints(0, 3) === 3` + `(2, 3) === 3` |
| [5] Server-null + chain present | `(null, 3) === 3` + `(undefined, 3) === 3` |
| [6] Sanitization with new chain-trust semantics | rebalanced to verify chain wins over server NaN/negative |
| [12] Source-grep — REST + WS + fight-room | 4 assertions pinning every serializer/push site |

Plus existing [2] Race A coverage unchanged — `(3, 0) === 0` still
holds (chain=0 → return 0, safe).

### Known parity gap (not in scope of this fix)

The DRAW path (`fight-room.ts:790-832`) uses optimistic `applyXp` +
fire-and-forget `update_after_fight_draw` without awaiting the chain
result, so it doesn't have an obvious place to push fresh
`character_data` post-chain-confirm. A draw that causes a level-up
would have the same staleness pattern. Currently it doesn't surface
the bug because the helper fix (#1 above) trusts chain regardless of
server-side staleness. A future refactor could await the draw
chain call symmetrically with the win/loss path, but doing so is a
separate change.
