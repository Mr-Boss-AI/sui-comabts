## 1. Session win — phantom-empty-kiosk closed (live-verified)

### Bug

ShakaLiX listed Pendant of Wrath for 0.2 SUI, MrBoss bought for 0.205 SUI,
NFT transferred — but seller UI showed `Profits = 0` and `Withdraw`
disabled. On-chain truth: ShakaLiX owned **two** `KioskOwnerCap`s. The
sale settled in Kiosk B (`0xf1c72097…8713f`, 0.2 SUI profits); the UI was
rendering Kiosk A (`0x6eae68f8…042cb`, empty). 0.2 SUI stranded.

Root cause: `marketplace::create_player_kiosk`
(`contracts/sources/marketplace.move:59`) is unconditional on-chain, so a
duplicate click mints a second cap. Frontend `useKiosk` resolved
`caps[0]` — `listOwnedObjects` order isn't stable, so the UI flip-flopped
between kiosks across sessions.

### Fix — `9441d2c`, four layers

1. **`useMarketplaceActions.createKiosk` pre-flight** — queries
   `KioskOwnerCap` ownership before signing; refuses dup with toast
   "You already own a Kiosk — refreshing.".
2. **`useKiosk` rewrite** — enumerates every cap, fetches every kiosk,
   aggregates profits / listings / item counts. Primary = first
   non-empty kiosk; pure aggregation extracted as `aggregateKiosks()`.
   Exposes `capForKiosk(kioskId)` lookup.
3. **`buildWithdrawAllKioskProfitsTx` + `withdrawAllProfits`** — single
   PTB sweeps every kiosk's profits, batches every `Coin<SUI>` into one
   `transferObjects`. Stranded SUI in orphan kiosks clears in one click.
4. **Per-item routing** — `Item.kioskId` stamped by `fetchKioskItems`;
   `handleDelist` / `handleRetrieveStuck` look up cap via
   `kiosk.capForKiosk(listing.kioskId)`. Ops always hit the kiosk that
   physically holds the item.

## 2. Three polish items uncovered tonight (NOT bugs, deferred)

### A. Empty primary kiosk after a fully-cleared session

`aggregateKiosks` picks the first non-empty kiosk as primary, and falls
back to the first cap in input order when every kiosk is empty. After
ShakaLiX withdrew everything, both kiosks went empty and the primary
landed back on `0x6eae68…042cb` (the original empty one). Funds are
always recoverable via the sweep PTB, so it's cosmetic — but the UI
shows the "wrong" kiosk ID in the footer when the user expects the
"newer" one.

**Fix shape (next session):** add a tiebreaker for the all-empty case.
Cheapest option: prefer the kiosk whose cap was minted most recently
(read the cap object's `previous_transaction` or its position in
`listOwnedObjects` if Mysten's RPC documents that as creation-time
ordering). More general option: hide an empty secondary kiosk from the
footer and show "You have N kiosks (1 active)" as a hint. ~30 min.

### B. "Create my Kiosk" button doesn't auto-hide after success

The button shows the success toast and the on-chain refresh triggers,
but the panel keeps the create CTA visible until the cap fetch
resolves. A second click is now blocked by the dup-create pre-flight
(`9441d2c`), so there's no harm — just a confusing flash.

**Fix shape (next session):** flip a local `creating` flag to true on
submit, keep it true until `kiosk.loaded && kiosk.kiosks.length > 0`,
render the loading shell while it's true. ~10 min.

### C. Mid-session re-auth popup during a buy

MrBoss saw a brief "Signing you in…" loader appear once during his
buy flow. Likely the zkLogin Enoki JWT expired mid-tx; the dapp-kit
session-restore kicked in transparently. No money lost, no tx failed,
but the UX flicker is undesirable on the marketplace.

**Fix shape (next session, or punt to a session-lifecycle bucket):**
instrument the zkLogin session lifetime — log token issuance time +
expiry on auth_ok, log every silent refresh, and surface a proactive
refresh ≤2 min before expiry. May want a `withZkLoginRefresh(...)`
wrapper similar to the `withTimeout(...)` pattern enforced by
`qa-messaging-client.ts`. Not a market bug — applies to every
`signAndExecuteTransaction` call site.

## 4. Mainnet-prep flag (v5.1 blocker)

`MAINNET_PREP.md` Section C Contract layer already covers this — pinning
again for visibility: the JS-side dup-kiosk guard shipped in `9441d2c` is
a **testnet bridge**. A hand-crafted PTB or two-tab race during
indexing lag can still bypass it. The proper fix for v5.1 republish:

```move
public struct KioskRegistry has key, store {
    id: UID,
    table: Table<address, ID>,
}

public fun create_or_get_player_kiosk(
    registry: &mut KioskRegistry,
    ctx: &mut TxContext,
): ID { ... }
```

`create_or_get_player_kiosk` returns the existing kiosk_id when sender is
already registered, otherwise creates one and inserts into the table.
Frontend drops the JS pre-flight when this lands.
