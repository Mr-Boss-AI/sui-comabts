## Reality-check on scope (transparent up-front)

I'm going to be honest here so the journal is trustworthy. The audit I wrote earlier today (which the user cites as PRIMARY SPEC) explicitly estimates v5.1 at **5-7 days of focused work + 2-4 weeks audit** before mainnet. That estimate is realistic.

Doing all of phases 1-8 in one overnight session at "holy shit, that's done" quality is not physically achievable. I will not pretend otherwise. The right approach is:

1. **Prioritize correctness over completeness.** Half-broken contracts on testnet are worse than no contracts.
2. **Prioritize reversibility.** Every commit goes to `feature/v5.1-contracts`. The branch can be discarded if the work isn't useful.
3. **Phase 1 (Move contracts + tests) gets the most care.** It's the foundation; everything else stacks on it.
4. **Phase 2 deploy goes ahead if Phase 1 is clean.** 3.47 SUI in TREASURY is enough.
5. **Phases 3-7 get as much as I can give them with honesty about gaps.** Critical mainnet-blocker work (settlement journal, confirm-modal gates) gets prioritised over polish.
6. **Decisions documented in this journal with reasoning.** Game-balance numbers are tentative; the user reviews when awake.

**What I will NOT do unilaterally:**
- Mint v5.1 NFTs to player wallets without confirming v5.0 → v5.1 player-wallet policy (the audit says fresh chain = no migration, but transferring fresh NFTs to specific wallets is a decision the user should make)
- Push commits that aren't behind cleanly-named conventional-commit messages
- Touch `main` or `feature/phase-2-design` branches
- Tear up `assertTxSucceeded` defaults for live frontend without paired tests
- Implement sui::random for combat (substantial architectural change; scope it to LOOT-only — high trust delta, low refactor cost)
