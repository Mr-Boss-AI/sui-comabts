# Two-handed weapon flow ‚Äî end-to-end (v5.1)

Closes `project_slot_type_seed.md`. Authored 2026-05-29 alongside the
Nail Plank regression fix.

## TL;DR

A two-handed weapon (greatsword, maul, war-pick, "Nail Plank", ‚Ä¶)
**occupies both the weapon slot and the offhand slot**. The
constraint is enforced in three places:

| Layer | Where | Mechanism |
|---|---|---|
| **Chain (source of truth)** | `contracts/sources/equipment.move` | `equip_weapon` / `equip_offhand` assertions on `Item.slot_type` |
| **Frontend logic** | `lib/two-handed-weapons.ts`, `lib/loadout-tx.ts`, `hooks/useEquipmentActions.ts` | Picker lock, classifier, save-time auto-reconcile, PTB op ordering |
| **Frontend UI ‚Äî passive** | `components/character/character-profile.tsx` | Off-hand `SlotTile` disabled when `pending.weapon` is 2H |
| **Frontend UI ‚Äî inventory panel** | `lib/equipment-picker.ts::getEquipTargetsForItem`, `components/items/inventory.tsx` | "EQUIP TO:" rows; off-hand row stripped for 2H weapons |
| **Frontend UI ‚Äî educational** | `components/character/two-handed-conflict-modal.tsx` | Center modal fires only on a *wrong-order* stage attempt |

The chain rule is the trustless backstop ‚Äî every frontend guard
mirrors a chain assertion, never goes beyond it.

## Data flow ‚Äî `slot_type` from chain to UI

`Item.slot_type: u8` is stamped at mint time in
`contracts/sources/item.move::mint_item_admin` and validated against
the item's `item_type` (weapons MUST be `SLOT_MAINHAND` or
`SLOT_BOTH_HANDS`; shields MUST be `SLOT_OFFHAND`; armor/jewellery
MUST be `SLOT_MAINHAND` by convention). The chain field then flows
through:

```
chain Item.slot_type (u8)
  ‚îÇ
  ‚îú‚îÄ‚îÄ server/src/utils/sui-read.ts::parseItemFromContent ‚Üí Item.slotType
  ‚îú‚îÄ‚îÄ server/src/data/marketplace.ts::fetchItemNft       ‚Üí Item.slotType
  ‚îÇ     ‚îÇ
  ‚îÇ     ‚îî‚îÄ‚îÄ server/src/utils/wire-sanitize.ts::sanitizeItem (carries slotType)
  ‚îÇ           ‚îÇ
  ‚îÇ           ‚îî‚îÄ‚îÄ frontend Item.slotType  (wire-borne inventory / scout)
  ‚îÇ
  ‚îî‚îÄ‚îÄ frontend/src/lib/sui-contracts.ts::fetchOwnedItems / fetchKioskItems
            ‚îî‚îÄ‚îÄ frontend Item.slotType  (wallet-side direct chain reads)
```

The frontend `Item.slotType` field is optional (`SlotType | undefined`)
because **legacy server-only NPC items** (UUID ids, generated by
`server/src/game/loot.ts`) predate chain enforcement. `undefined` is
treated as `MAINHAND` everywhere ‚Äî the only sane default for non-weapon
armor, and NPC loot never includes two-handed weapons anyway.

The constants are mirrored in `frontend/src/types/game.ts::SLOT_TYPES`:

```ts
export const SLOT_TYPES = {
  MAINHAND: 0,      // single-handed weapon ‚Äî weapon slot only
  OFFHAND: 1,       // shield or off-hand weapon ‚Äî offhand slot only
  BOTH_HANDS: 2,    // two-handed ‚Äî weapon slot + requires offhand empty
} as const;
```

## Chain enforcement ‚Äî abort codes 6 / 7 / 8 / 9

`contracts/sources/equipment.move` consumes `slot_type` in three asserts:

| Code | Constant | Site | Condition |
|---|---|---|---|
| 6 | `EOffhandOccupied` | `equip_weapon:68` | 2H weapon attempted while offhand DOF is non-empty |
| 7 | `EWeaponIsTwoHanded` | `equip_offhand:109` | offhand attempted while current weapon is 2H |
| 8 | `EItemNotMainhand` | `equip_weapon:65` | weapon slot got an offhand-typed item (shield) |
| 9 | `EItemNotOffhand` | `equip_offhand:98,100` | offhand slot got a 2H weapon, or a shield without slot_type=OFFHAND |

These are correct rejections ‚Äî the chain is the source of truth. The
frontend layers below exist to keep users from triggering them, and to
translate them into plain English if one slips through.

## Frontend stage classifier ‚Äî `classifyStageEquip`

The decision helper that drives the three UX paths in
`useEquipmentActions::stageEquip`. Pure, no React, no dispatch ‚Äî
pinned by `scripts/qa-two-handed-stage-classifier.ts`.

| Decision | Trigger | UX response |
|---|---|---|
| `auto_clear` | 2H weapon staged into weapon slot AND off-hand currently occupied | Silent `STAGE_UNEQUIP("offhand")` + bottom toast *"Off-hand removed ‚Äî two-handed weapon equipped."* No modal. |
| `block_and_explain` | Off-hand attempt that conflicts with 2H (2H into off-hand slot, OR any off-hand item while `pending.weapon` is 2H) | Refuse the stage **and** dispatch `SHOW_TWO_HANDED_CONFLICT_MODAL` ‚Äî the educational center modal opens. |
| `ok` | Anything else, including the **correct** unequip-first flow (off-hand cleared before equipping 2H, or 2H cleared before equipping off-hand) | Proceed normally. **No modal, no toast.** |

The classifier is the single point where the modal is triggered. The
correct-order flow is `'ok'` by construction ‚Äî staging an off-hand
when `pending.weapon` is null skips both conflict cases. Players who
learn the rule stop hitting `block_and_explain`, the modal stops
firing. **Self-extinguishing ‚Äî no localStorage seen-it flag needed.**

## Frontend layer 0 ‚Äî inventory "EQUIP TO:" panel

`lib/equipment-picker.ts::getEquipTargetsForItem(item)` returns the
list of slot rows the inventory item-detail panel should render.
`SLOT_TO_ITEM_TYPE.offhand` includes WEAPON because dual-wield is a
real config; the helper post-filters: a weapon with
`slotType === BOTH_HANDS` never returns `'offhand'`. Pinned by
`qa-equip-picker.ts` block [12.5] (9 assertions covering 1H weapons,
all 3 catalog greatswords, the Nail Plank case, shields, legacy NPC
weapons, and unrelated slots).

The 2026-05-29 *"Nail Plank picker still shows Off-hand"* report
turned out to be **server-side staleness** ‚Äî `ts-node` doesn't watch,
so the running server was emitting wire payloads without `slotType`,
falling into the legacy-NPC-compat branch (returns both rows). Fix:
restart the server. Pinned by the gauntlet which constructs items
with explicit `slotType: SLOT_TYPES.BOTH_HANDS`.

## Frontend layer 1 ‚Äî picker prevention

`frontend/src/lib/equipment-picker.ts::buildSlotPickerEntries` consults
`evaluateTwoHandedConflict` (in `lib/two-handed-weapons.ts`) when
building the offered-item list per slot. The three blocked cases:

1. **2H weapon offered for offhand slot** ‚Äî locked with reason
   *"Two-handed weapon ‚Äî equip in the weapon slot."*
2. **Anything offered for offhand while pending.weapon is 2H** ‚Äî
   locked with reason
   *"Two-handed weapon equipped ‚Äî unequip it before adding an off-hand."*
3. **2H weapon offered for weapon slot while pending.offhand is
   occupied** ‚Äî surfaced as **informational** rather than blocking,
   with reason *"Two-handed weapon ‚Äî equipping will clear your
   off-hand."* The user can still click through; the save-time
   auto-reconcile (layer 3) does the actual offhand clear.

`isTwoHanded()` and `evaluateTwoHandedConflict()` both read
`item.slotType`, **not** the item's name. The pre-v5.1
`TWO_HANDED_NAMES` allowlist is deleted.

## Frontend layer 2 ‚Äî UI slot lock

`components/character/character-profile.tsx::EquipmentFrame` computes:

```ts
const offhandLocked = eq.weapon != null && isTwoHanded(eq.weapon);
```

‚Ä¶and forwards `disabled={offhandLocked}` plus a tooltip to the off-hand
`SlotTile`. The slot dims to 40% opacity, the cursor becomes
`not-allowed`, and the click handler is bypassed. The user gets the
message **before** they spend a click trying to stage an off-hand
under a 2H weapon ‚Äî closes the silent-failure gap.

## Frontend layer 3 ‚Äî save-time invariant + PTB ordering

`frontend/src/lib/loadout-tx.ts::buildSaveLoadoutTx` is the
defence-in-depth backstop. It enforces two rules at PTB-build time:

### Rule A ‚Äî 2H + offhand reconciliation

If `pending.weapon` is two-handed AND `pending.offhand` is non-null
(layers 1 & 2 should already prevent this ‚Äî this is the race / direct
state-mutation guard), the build silently clears the offhand in a
defensive copy:

```ts
let reconciledPending = pending;
let offhandAutoCleared = false;
if (pending.weapon && isTwoHanded(pending.weapon) && pending.offhand) {
  reconciledPending = { ...pending, offhand: null };
  offhandAutoCleared = true;
}
```

The caller (`saveLoadout` in `useEquipmentActions.ts`) **must** rebase
`committed := reconciledPending` on success, **not** `committed :=
pending`. The build returns `offhandAutoCleared: true` so the caller
can surface a non-blocking toast (*"Off-hand removed ‚Äî two-handed
weapon equipped."*).

### Rule B ‚Äî PTB op ordering (unequip before equip)

The chain checks slot-emptiness assertions per-command, in PTB order.
A na√Øve per-slot loop `unequip_<slot>; equip_<slot>` is fine within
one slot, but breaks for **cross-slot** invariants like the 2H rule.
Concrete failure case:

- `committed = { weapon: Longsword, offhand: Shield }`
- `pending  = { weapon: NailPlank(2H), offhand: null }`

Na√Øve PTB:
```
unequip_weapon   ‚Üê Longsword out
equip_weapon(NailPlank)  ‚Üê FAILS: offhand DOF still holds Shield ‚Üí EOffhandOccupied
unequip_offhand  ‚Üê would have cleared Shield, but PTB already aborted
```

The fix is a **two-phase build**: all `unequip_*` calls first, then
all `equip_*` calls, then `save_loadout`. The same example becomes:

```
Phase 1:
  unequip_weapon   ‚Üê Longsword out
  unequip_offhand  ‚Üê Shield out
Phase 2:
  equip_weapon(NailPlank)  ‚Üê offhand DOF empty ‚Üí passes
Phase 3:
  save_loadout
```

The rule is general: **ops that free slots run before ops that occupy
conflicting ones**. The phase split also keeps the PTB deterministic +
stable across renders (useful for digest dedupe + log parsing).

## Frontend layer 4 ‚Äî abort-code translation

If a chain abort somehow slips through (older client, programmatic
PTB build, race condition, etc.), the catch block in
`hooks/useEquipmentActions.ts::saveLoadout` translates the raw
`MoveAbort` string into plain English via `humanizeChainError(raw,
EQUIPMENT_ABORT_CODES)`. The mapping table lives in
`lib/equipment-aborts.ts` and pins to the Move constants:

```ts
6 ‚Üí "That's a two-handed weapon ‚Äî unequip your off-hand first."
7 ‚Üí "You're holding a two-handed weapon. Unequip it to use an off-hand."
8 ‚Üí "That item can't go in the weapon slot ‚Äî it's an off-hand item."
9 ‚Üí "That item can't go in the off-hand slot ‚Äî it's a two-handed or main-hand-only item."
```

### The dapp-kit 2.16 throw caveat

The `@mysten/dapp-kit-core` 2.16 SDK **throws** an `Error` directly on
chain abort rather than resolving the signer promise with
`$kind=FailedTransaction`. Consequence: `assertTxSucceeded` ‚Äî which
holds the abort-code map and humanizes ‚Äî **never runs** for those
aborts. The catch block is where the SDK-thrown error actually lands,
so it **must** pass the map to `humanizeChainError` itself, or the
user sees the generic `"Abort code 6"` fallback. This was the
2026-05-29 bug; fixed in `useEquipmentActions.ts:saveLoadout` (and the
parallel wager paths in `matchmaking-queue.tsx`).

## Adding a new two-handed weapon

Mint it on chain with `slot_type = SLOT_BOTH_HANDS`. No frontend
change required ‚Äî the picker, lock, diff, and toast all derive from
the chain field.
